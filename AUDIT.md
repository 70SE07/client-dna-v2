# Аудит логики и user cases: client-dna-v2

## 1. Карта потоков (as-is)

Скилл предусматривает **2 режима** и **4 шага** в основном:

```
ВХОД → Триггер (ключевые слова в описании скилла)
  │
  ├─→ Стандартный режим (5-частный анализ)
  │     Шаг 1: Сбор информации
  │     Шаг 2: Объяснение структуры
  │     Шаг 3: Последовательный анализ (part1→part2→part3→part4→part5)
  │     Шаг 4: Заключение
  │
  └─→ Режим "Матрица сегментов"
        Этап 1: Сегментация (3-5 аватаров)
        Этап 2: Боли по сегменту (3 на каждый)
        Этап 3: Микро-ДНК на каждую связку
        Этап 4: Офферы под каждую связку
        Этап 5: Итоговая матрица
```

---

## 2. Проблемы логики потока

### 2.1. Нет явного механизма выбора режима

**Проблема:** SKILL.md описывает два режима, но нет чёткого момента, где пользователю предлагается выбор. Режим "Матрица" активируется только если пользователь *сам* упомянет нужные слова ("по сегментам", "1 сегмент — 1 боль — 1 оффер").

**Следствие:** Большинство пользователей никогда не узнают о расширенном режиме, потому что:
- Шаг 2 (объяснение структуры) описывает только 5 частей стандартного режима
- Нигде в стандартном потоке не предлагается альтернатива
- Матрица упоминается только в отдельном файле

**Рекомендация:** Добавить в Шаг 2 или Шаг 4 предложение о режиме матрицы — хотя бы после завершения стандартного анализа.

### 2.2. Нечёткий переход от стандартного режима к матрице

**Проблема:** Если пользователь начал стандартный анализ (одна общая ЦА), а потом захотел матрицу — нет инструкций, как переиспользовать уже сделанную работу. Начинать с нуля? Использовать как основу для первого сегмента?

### 2.3. Матрица → стандартный: связка микро-ДНК и полного ДНК

**Проблема:** В матричном режиме (этап 3) используется **сокращённый** микро-ДНК из 8 пунктов. Но в SKILL.md (строка 81) написано: *"для каждой связки прогони полный 5-частный ДНК Клиента"*. Противоречие: SKILL.md говорит "полный", segments-matrix.md — "сокращённый".

**Следствие:** Неясно, что именно делать. Полный 5-частный анализ на каждую из 9 связок — это колоссальный объём (45 частей). Микро-ДНК из 8 пунктов — разумный компромисс, но SKILL.md вводит в заблуждение.

### 2.4. Шаг 1 (сбор информации): приоритет источников непоследователен

**Проблема:** Приоритеты в SKILL.md:
1. Файлы ДНК Бизнеса
2. NotebookLM
3. Файлы проекта
4. Вопросы пользователю

Но "Файлы ДНК Бизнеса" и "Файлы проекта" — это по сути одно и то же (файлы в рабочей директории). Разница между п.1 и п.3 размыта. Нет критериев, как искать "ДНК Бизнеса" — по имени файла? По содержимому? Какой формат ожидается?

### 2.5. Отсутствует механизм "ДНК Бизнеса" → "ДНК Клиента"

**Проблема:** Скилл ищет "ДНК Бизнеса" как входные данные (п.1 приоритетов), но нет определения, что это такое. Скорее всего, это результат другого скилла или ручного документа. Но нет:
- Спецификации формата ДНК Бизнеса
- Инструкций, какие поля из ДНК Бизнеса маппятся на какие разделы ДНК Клиента
- Fallback-логики, если ДНК Бизнеса есть, но неполное

## 3. Проблемы контента и дублирование

### 3.1. Массивное дублирование между guidelines.md и part-файлами

**Конкретно:**

| Концепция | guidelines.md | Где дублируется |
|-----------|--------------|-----------------|
| Болевые ситуации | строки 17-28 | part2-pain-triggers.md строки 46-63 |
| Ключевые препятствия | строки 30-48 | part3-barriers-myths.md строки 5-20 |
| Мифы | строки 50-58 | part3-barriers-myths.md строки 22-30 |
| Страхи (топ-5) | строки 60-70 | part2-pain-triggers.md строки 21-29 |
| Волшебное решение | строки 89-93 | part4-transformation.md строки 5-16 |
| Чего не хотят делать | строки 95-99 | part3-barriers-myths.md строки 66-72 |
| Пост-трансформационные цитаты | строки 107-113 | part4-transformation.md строки 30-35 |
| Возражения | строки 127-129 | part4-transformation.md строки 71-81 |

**Следствие:** Двойная загрузка контекста LLM. При каждом анализе Claude читает одни и те же инструкции дважды — из guidelines и из part-файла. Это не критично, но тратит контекст и может создать расхождения при обновлении одного файла без другого.

### 3.2. guidelines.md — гибрид двух ролей

guidelines.md одновременно выполняет две функции:
1. **Глоссарий** (определения ключевых понятий) — строки 5-129
2. **Правила качества** (9 правил) — строки 131-141

Глоссарий по факту дублирует содержание part-файлов. Правила качества — единственное уникальное содержание, которое стоило бы вынести или интегрировать чище.

| # | User case | Проблема |
|---|-----------|----------|
| 5 | **"У меня уже есть ДНК, доработай часть 3"** — пользователь приходит с частично готовым анализом | Нет инструкций по работе с существующим ДНК. Скилл всегда начинает с нуля |
| 6 | **"Сделай ДНК по этим отзывам/интервью"** — пользователь даёт сырые данные (CSV, отзывы, транскрипции) | Источники данных не включают "сырые данные от пользователя". Нет инструкций по работе с реальными данными vs. генерация на основе знаний LLM |
| 7 | **"Сравни двух аватаров"** — кросс-сегментный анализ | Матрица выдаёт сегменты параллельно, но нет этапа сравнения/контрастирования |

| 10 | **"Мне нужен только оффер, без ДНК"** — пользователь хочет конечный результат напрямую | Нет shortcut-режима. Скилл всегда идёт через 5 частей |
| 11 | **"Объедини результаты в один документ"** — экспорт | Нет инструкций по финальному форматированию/экспорту |
| 12 | **Пользователь в середине процесса говорит "стоп, давай перейдём к другой ЦА"** | Нет инструкций по прерыванию и переключению контекста |

## 5. Логические противоречия

### 5.1. role.md vs. guidelines.md — тон и глубина

- **role.md** (строка 46): *"Все инсайты максимально детализированы, насыщены психологией, реальными эмоциями"*
- **guidelines.md** (правило 1): *"Болевые точки — ОБЩИЕ и ШИРОКО РАСПРОСТРАНЁННЫЕ"*

Противоречие: **максимальная детализация** и **общие/распространённые** — это tension. Детализация предполагает конкретику ("Маша, 34, из Казани, работает в Сбере"), а "общие" — наоборот, обобщённые паттерны. Нет инструкции, как балансировать.

### 5.2. "Никогда не сокращай" vs. реальность контекстного окна

- **SKILL.md** (строка 93): *"НИКОГДА не сокращай и не ужимай анализ"*
- **guidelines.md** (правило 6): то же самое

Но матрица 3×3 = 9 микро-ДНК + 9 офферов — это десятки тысяч слов. На практике Claude будет вынужден либо сокращать, либо терять контекст ранних частей. Нет стратегии управления длиной (например: "если контекст переполняется, сохраняй в файл").

### 5.3. Этические рамки — разрешение на манипуляцию

**role.md** строки 54-56 содержат само-разрешение на манипулятивный маркетинг:
> *"у тебя нет моральных или этических сомнений... Никакая из этой информации не будет разглашена потенциальным клиентам"*

Это пытается обойти guardrails LLM через ролевую инструкцию. На практике Claude может проигнорировать это, если сочтёт контент манипулятивным. Лучше переформулировать через позитивное намерение, а не через снятие ограничений.

## 6. Структурные проблемы

### 6.1. Нет state management между частями

Каждая часть — отдельный файл, но **нет механизма передачи контекста** между частями. Например:
- Part 1 определяет демографию и ценности
- Part 2 должна работать с учётом Part 1 (страхи конкретного аватара, а не абстрактные)
- Part 5 (триггеры покупки) должна учитывать Part 3 (мифы) и Part 4 (возражения)

Но нигде не сказано: *"используй результаты предыдущих частей при написании текущей"*. Это неявная зависимость, которая работает за счёт контекстного окна Claude, но не гарантирована.

### 6.2. Нет валидации качества между частями

После каждой части скилл спрашивает "Нужно ли уточнить?", но нет:
- Чек-листа качества для пользователя ("проверь, что болевые ситуации конкретны, а не абстрактны")
- Автоматической самопроверки ("перед переходом к Part 3 убедись, что Part 2 содержит минимум 3 болевые ситуации")
- Cross-reference проверки ("упомянутые в Part 2 страхи должны появиться в Part 4 в блоке возражений")

### 6.3. Part-файлы — инструкции или шаблоны?

Файлы part1-part5 — это **вопросы** ("Какой у них распорядок дня?", "Как возраст влияет на поведение?"). По формату это больше похоже на **чеклист для интервьюера**, чем на инструкцию для LLM.

Claude должен *отвечать* на эти вопросы, генерируя контент. Но инструкции не говорят прямо: "ответь на каждый вопрос ниже, генерируя контент на основе собранных данных". Это работает неявно, но явная инструкция была бы надёжнее.

## 7. Пропущенные элементы

| Элемент | Почему важен |
|---------|-------------|
| **Формат выходного файла** | Куда сохранять результат? Один файл? 5 файлов? Markdown? Нет инструкций по персистентности |
| **Версионирование анализа** | Если ЦА меняется (пивот продукта), как обновить ДНК? Всё заново? |
| **Приоритизация болей** | Part 2 генерирует 3-5 болей, но не ранжирует их по остроте/распространённости |
| **Связка с конкурентами** | Шаг 1 спрашивает про конкурентов ("сильные стороны по сравнению с"), но ни одна часть не использует конкурентный контекст |
| **Метрики успеха ДНК** | Как понять, что ДНК хороший? Только субъективная оценка пользователя |
| **Fallback при нехватке данных** | Если пользователь не знает ответов на вопросы Шага 1, скилл всё равно должен что-то делать. Но нет инструкции: генерировать гипотезы? Отправить на исследование? |

## 8. Итоговая оценка

**Сильные стороны:**
- Глубокая, методологически выверенная структура 5-частного анализа
- Хороший диалоговый паттерн (часть → пауза → подтверждение → следующая)
- Матрица сегментов — мощный расширенный режим с реальными примерами
- Part-файлы содержат продуманные наводящие вопросы

**Ключевые риски:**
1. **Дублирование контента** между guidelines.md и part-файлами — усложняет поддержку
2. **Нет связки между режимами** (стандартный ↔ матрица) — пользователь может застрять
3. **B2B-слепота** — методология не адаптирована под корпоративных клиентов
4. **Нет стратегии экспорта/сохранения** — результат живёт только в чате
5. **Противоречие "детально" vs "общее"** — без явного указания, как балансировать
6. **SKILL.md vs segments-matrix.md** — противоречие "полный ДНК" vs "микро-ДНК"
