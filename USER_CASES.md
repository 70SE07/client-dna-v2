# User Cases: ДНК Клиента

---

## UC-1: Полный стандартный анализ

**Пользователь:** Маркетолог или предприниматель, который знает свой продукт и хотя бы примерно понимает свою ЦА.

**Запрос:** "Сделай ДНК клиента для моего онлайн-курса по SMM"

**Что имеет на руках:**
- Знает свой продукт
- Имеет представление о ЦА (пусть даже поверхностное)
- Может ответить на вопросы о болях, конкурентах, сильных сторонах

**Что получает:**
- Документ-фундамент: портрет ЦА, боли, барьеры, трансформация, триггеры + оффер(ы)

**Поток:**
```
Пользователь описывает продукт/ЦА → Система уточняет → 5-частный анализ → Оффер
```

### Заметки

Этот кейс покрыт текущей методологией полностью. Шаг 1 (сбор информации) работает, потому что пользователь способен ответить на вопросы. Part 1-5 прогоняются последовательно, на выходе — оффер.

## UC-2: Углубление конкретной части

**Пользователь:** Тот же, кто прошёл UC-1 (или UC-3, UC-4, UC-5). Получил оффер, но чувствует, что какая-то "нога" шатается.

**Запрос:** "Расскажи подробнее про страхи" / "Боли слишком поверхностные, давай перекопаем Part 2"

**Что имеет на руках:**
- Готовый документ ДНК (результат предыдущего анализа)
- Понимание, что именно не устраивает

**Что получает:**
- Доработанная часть анализа → пересобранный оффер

**Поток:**
```
Пользователь указывает на слабое место → Система открывает конкретную часть → Доработка → Пересборка оффера
```

### Заметки

Этот кейс — реализация гибридного режима, который мы обсуждали. Пользователь всегда начинает с результата (оффер), а глубина — по необходимости. Не нужно выбирать режим заранее.

Оффер стоит на 5 ногах, пользователь может покопать любую:
1. **Кто этот человек (Part 1)** — может неправильно определили кто клиент
2. **Что у него болит (Part 2)** — боли слишком общие, нужна конкретика
3. **Почему он застрял (Part 3)** — неправильно поняли причину, почему не покупает
4. **Что он хочет получить (Part 4)** — обещаем не то, что клиент реально хочет
5. **Что толкает купить сейчас (Part 5)** — давим не на тот триггер

Покопал нужную ногу → оффер пересобрался точнее.

Для реализации: результат каждой части должен сохраняться в файл, чтобы при возврате (даже в новом чате) контекст не терялся.

## UC-3: Матрица сегментов

**Пользователь:** Маркетолог, который понимает что ЦА неоднородна и хочет несколько офферов под разные сегменты.

**Запрос:** "Сделай анализ по сегментам" / "1 сегмент — 1 боль — 1 оффер"

**Что имеет на руках:**
- Знает свой продукт
- Понимает что аудитория разная

**Что получает:**
- Матрица: N сегментов × 3 боли = N×3 микро-ДНК + N×3 офферов
- Каждый оффер привязан к конкретному сегменту и конкретной боли

**Поток:**
```
Сбор данных → Сегментация (3-5 аватаров) → Боли по сегменту → Микро-ДНК на связку → Оффер на связку → Итоговая матрица
```

### Заметки

Этот кейс покрыт текущей методологией (segments-matrix.md). Но есть проблема обнаруживаемости: пользователь узнает о матричном режиме только если сам упомянет нужные слова. Нигде в стандартном потоке (UC-1) не предлагается альтернатива.

Рекомендация: после завершения UC-1 (стандартный анализ) предлагать пользователю: "Хочешь разбить ЦА на сегменты и получить отдельный оффер под каждый?"

Также есть противоречие в SKILL.md: написано "прогони полный 5-частный ДНК Клиента" на каждую связку, а segments-matrix.md описывает сокращённый микро-ДНК из 8 пунктов. Нужно привести к единому пониманию.

## UC-4: Старт с нуля — "ничего не знаю о ЦА"

**Пользователь:** Предприниматель с идеей, но без данных. Не изучал ЦА, не знает боли, не проводил кастдевы.

**Запрос:** "Хочу сделать продукт — трудоустройство за границей. Ничего не знаю о ЦА. Помоги собрать всё, чтобы у меня был документ с офферами, болями, ЦА и я мог протестировать гипотезу."

**Что имеет на руках:**
- Только тему/нишу ("трудоустройство за границей")
- Ноль данных о ЦА
- Цель — получить документ-фундамент, чтобы запустить тест (сайт + трафик) и проверить есть ли спрос

**Что получает:**
- Тот же документ-фундамент что и в UC-1, но маркированный как **гипотеза**
- Оффер(ы) для тестирования

**Поток:**
```
Пользователь называет тему → Система генерирует гипотетические сегменты → Пользователь выбирает → Стандартный пайплайн → Оффер (маркирован как гипотеза)
```

### Заметки

**Дыра 1: Шаг 1 должен уметь работать с "ничего не знаю".**

Сейчас: вопросы предполагают знание → пользователь застревает. Все вопросы Шага 1 ("Для кого продукт предназначен?", "Какие основные проблемы решает?", "Что известно о боли ЦА?") ожидают, что пользователь ЗНАЕТ ответы. Если он на всё отвечает "не знаю" — скилл не имеет инструкций, что делать.

Нужно: если пользователь не знает ЦА — система переключается в режим "давай вместе определим, кто твой клиент". Берёт тему ("трудоустройство за границей"), генерирует 3-5 гипотетических сегментов, показывает пользователю, спрашивает "кто из них ближе к тому, что ты хочешь делать?". После выбора — запускает стандартный пайплайн.

**Дыра 2: Результат должен быть маркирован — это гипотеза или анализ.**

Если у пользователя были реальные данные (кастдевы, отзывы) → документ маркируется как "анализ на основе данных".
Если данных не было → документ маркируется как "гипотеза, требует проверки". Это честно и это важно, потому что пользователь сам сказал — он хочет "протестировать гипотезу и сказать да спрос есть или нет".

Ядро методологии (Part 1-5) при этом работает: Claude знает рынки и может сгенерировать качественную гипотезу по ЦА, болям и офферам даже с нуля. Нужно только научить Шаг 1 работать с пустым входом.

## UC-5: Доработка существующего ДНК

**Пользователь:** Тот, кто уже проходил анализ раньше. Продукт изменился, ЦА сместилась, или результаты теста показали, что гипотеза не сработала.

**Запрос:** "У меня уже есть ДНК, нужно обновить" / "Протестировал — оффер не зашёл, давай разберёмся"

**Что имеет на руках:**
- Готовый документ ДНК (файлы с прошлого анализа)
- Новые данные или обратная связь с рынка

**Что получает:**
- Обновлённый документ с учётом новых данных → пересобранный оффер

**Поток:**
```
Пользователь указывает, что изменилось → Система загружает существующий ДНК → Доработка нужных частей → Пересборка оффера
```

### Заметки

Этот кейс НЕ покрыт текущей методологией. Нет инструкций по работе с существующим ДНК:
- Как загружать предыдущий результат?
- Как определить, какие части нужно обновить, а какие остаются?
- Как мержить новые данные с существующим анализом?

Это по сути цикл обратной связи: пользователь получил документ (UC-1 или UC-4), пошёл тестировать, вернулся с результатами и хочет итерировать. Система должна уметь принять его обратно с сохранённым контекстом.

Для реализации в MCP: если результаты сохраняются в файлы (storage), то при повторном входе система читает существующие файлы и предлагает: "У тебя уже есть ДНК. Что хочешь обновить?"

## UC-6: Анализ на основе сырых данных

**Пользователь:** Маркетолог, который провёл кастдевы, собрал отзывы, у него есть транскрипции интервью, скриншоты из чатов, CSV с ответами.

**Запрос:** "Вот транскрипции 10 интервью с клиентами. Собери ДНК на основе реальных данных."

**Что имеет на руках:**
- Сырые данные: тексты интервью, отзывы, ответы на анкеты
- Возможно — уже готовый продукт с реальными клиентами

**Что получает:**
- Документ-фундамент, основанный на реальных данных (маркирован как "анализ на основе данных")
- Оффер(ы), подкреплённые реальными цитатами и паттернами

**Поток:**
```
Пользователь загружает данные → Система анализирует → Извлекает паттерны → Стандартный пайплайн с реальными данными → Оффер
```

### Заметки

Этот кейс НЕ покрыт текущей методологией. Источники данных в SKILL.md (Шаг 1) не включают "сырые данные от пользователя". Нет инструкций:
- В каком формате принимать данные?
- Как извлекать паттерны из сырых текстов?
- Как отличать единичный случай от распространённого паттерна?
- Как использовать реальные цитаты клиентов вместо сгенерированных?

Это принципиально отличается от UC-1 (пользователь отвечает на вопросы устно) и UC-4 (данных нет вообще). Здесь система должна уметь работать с массивом неструктурированных данных и превращать их в структурированный ДНК.

При реализации в MCP: возможно отдельный tool `ingest_data`, который принимает сырые данные и превращает их в структурированный контекст для дальнейшего пайплайна.
